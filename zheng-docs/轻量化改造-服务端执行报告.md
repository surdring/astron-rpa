## 一、RPA 业务表创建（PostgreSQL 迁移脚本执行）

### 1.1 任务描述

执行 `backend/src/infra/database/migrations/001_create_rpa_tables.sql`，在 PostgreSQL 中创建 AstronRPA 核心业务表，配置 RLS 行级安全策略、索引和实时通知触发器。

### 1.2 执行步骤

1. 确认迁移脚本内容（8 张表，含 RLS、索引、触发器）
2. 检查 PostgreSQL 数据库连接（`insforge` 数据库）
3. 执行迁移脚本时发现 SQL 语法错误，修复后重新执行
4. 验证表结构、RLS 策略、触发器和索引

### 1.3 创建的数据库表

| 表名 | 说明 | 主要字段 |
|------|------|----------|
| `public.robots` | 机器人（RPA 客户端实例） | id, name, status, group_id, version, last_heartbeat, user_id |
| `public.projects` | 项目 | id, name, description, robot_id, user_id |
| `public.workflows` | 工作流 | id, project_id, name, definition, config, user_id |
| `public.tasks` | 任务 | id, workflow_id, status, priority, scheduled_at, user_id |
| `public.executions` | 执行记录 | id, task_id, robot_id, status, start_time, end_time, user_id |
| `public.execution_logs` | 执行日志 | id, execution_id, level, message, component_name, user_id |
| `public.market_components` | 市场组件 | id, name, type, version, publisher_id, description |
| `public.triggers` | 触发器 | id, name, type, config, enabled, user_id |

### 1.4 创建的 RLS 策略

- 所有包含 `user_id` 的表应用通用 RLS：`auth.uid() = user_id`
- `market_components` 表对所有用户可读，仅发布者可写
- 使用 `DROP POLICY IF EXISTS` + `CREATE POLICY` 而非 `CREATE POLICY IF NOT EXISTS`（避免 PostgreSQL 语法错误）

### 1.5 创建的触发器

- `public.set_user_id()` 函数：自动从 JWT 提取 `user_id` 填充到行
- 为所有业务表创建 `BEFORE INSERT` 触发器
- 为 `executions` 表创建 `execution_change_trigger` 实时通知触发器（`pg_notify`）

### 1.6 创建的索引

| 表 | 索引字段 |
|----|----------|
| robots | user_id, status, group_id |
| projects | user_id, robot_id |
| workflows | project_id, user_id |
| tasks | workflow_id, status, scheduled_at, user_id |
| executions | task_id, robot_id, status, user_id |
| execution_logs | execution_id, level, logged_at |
| market_components | type, publisher_id |
| triggers | user_id, enabled, type |

### 1.7 遇到的问题及解决

| 问题 | 原因 | 解决方式 |
|------|------|----------|
| `syntax error at or near "NOT"` | `CREATE POLICY IF NOT EXISTS` 语法在 PostgreSQL 18 中不支持 `IF NOT EXISTS` | 改为 `DROP POLICY IF EXISTS` + `CREATE POLICY` |
| `column "user_id" does not exist in "execution_logs"` | 迁移脚本中 `execution_logs` 表缺少 `user_id` 列定义 | 使用 `ALTER TABLE` 添加列，并更新迁移脚本 |

### 6.4 遇到的问题及解决

| 问题 | 原因 | 解决方式 |
|------|------|----------|
| `new row violates row-level security policy` | INSERT 时 `user_id` 为空，RLS 策略拒绝 | 添加 `set_user_id()` 触发器自动填充 |
| `Could not find the table 'public.robots' in the schema cache` | PostgREST schema 缓存未刷新 | 执行 `NOTIFY pgrst, 'reload schema'` |
| `Could not find the 'group_id' column of 'robots'` | robots 表无 group_id 列（迁移脚本未定义） | 移除查询中的 group_id 字段 |
| `Could not find the 'description' column of 'robots'` | 请求包含表中不存在的字段 | 移除 `description` 字段 |
| Edge Functions 返回 "Security Initialization Error" | Deno 2.9.1 中 `globalThis.Deno` 为不可配置（non-configurable），worker-template.js 尝试用 `Object.defineProperty` 重新定义导致错误 | 修改 worker-template.js：不再重新定义 `globalThis.Deno`，改为直接 patch `Deno.env` 的方法（`get/set/delete/toObject/has`） |
| Edge Functions 中 `client.auth.getCurrentUser()` 返回 "Unauthorized" | SDK 的 `getCurrentUser()` 需要用户 JWT 验证，但 Edge Function 上下文中仅使用 `anonKey` 认证 | 改用 `atob()` 直接从 JWT token 解析 `payload.sub` 获取用户 ID，避免调用 auth API |
| Edge Functions 查询 `notification_logs`/`blacklist` 表返回 "table not found" | 这两个表未在数据库中创建 | 在 PostgreSQL 中创建 `notification_logs` 和 `blacklist` 表，配置 RLS 策略 |
| Edge Functions 写入 `notification_logs` 表时 "new row violates row-level security policy" | RLS 策略拒绝 `anon` 角色的写入操作 | 创建 `allow_all_anon` 和 `allow_all_auth` 策略（`USING (true) WITH CHECK (true)`） |
| ai-service `/v1/models` 返回 404 | 该路由需特定认证头或配置 | 待确认 ai-service 路由配置 |
