## 一、背景与现状

### 当前架构问题

| 服务 | 数据库层 | 前端 API 层 | 引擎 API 层 | 业务逻辑层 |
|------|----------|-------------|-------------|-----------|
| **robot-service** | ✅ 表已创建（robots/tasks/executions/triggers） | ⏳ 部分迁移（CRUD 已迁移，复杂业务未迁移） | ❌ 仍依赖 | ❌ 未开始 |
| **resource-service** | ❌ 无对应表 | ❌ 仍依赖 | ❌ 仍依赖 | ❌ 未开始 |

### 核心原则

1. **引擎层先行** — scheduler 的 terminal 注册/心跳/执行上报可直接通过 `insforge_client.py` 已有方法迁移，无服务器依赖
2. **前端 CRUD 直接迁移到 InsForge SDK** — 简单的增删改查无需 Edge Functions
3. **复杂业务逻辑通过 Edge Functions 实现** — 需要多表联合查询、权限校验、事务性操作的逻辑，用 Deno TypeScript 编写 Edge Functions
4. **文件管理迁移到 InsForge Storage** — 替代 resource-service 的 S3 文件上传下载

#### 2.1.1 Terminal 注册与心跳

**当前文件**: [`terminal.py`](file:///d:/develop/astron-rpa-bak/engine/servers/astronverse-scheduler/src/astronverse/scheduler/core/terminal/terminal.py)

**当前调用**:
```python
# 注册: POST /api/robot/terminal/register
requests.post(f"{gateway}/api/robot/terminal/register", json={
    "terminalId": terminal_id, "name": name, "account": account,
    "os": os, "osPwd": os_pwd, "port": port, "ip": ip,
    "status": status, "cpu": cpu, "memory": memory, "disk": disk,
    "isDispatch": is_dispatch, "monitorUrl": monitor_url
})

# 心跳: POST /api/robot/terminal/beat
requests.post(f"{gateway}/api/robot/terminal/beat", json={
    "terminalId": terminal_id, "status": status,
    "isDispatch": is_dispatch, "cpu": cpu, "memory": memory, "disk": disk
})
```

**InsForge 替代方案**（`insforge_client.py` 已有方法）:

```python
from engine.shared.insforge_client import insforge

# 注册机器人
insforge.register_robot({
    "agent_key": terminal_id,          # 对应 terminalId
    "name": hostname,
    "host_name": hostname,
    "ip_address": ip,
    "os_version": os_info,
    "status": "online",
    "version": version,
})

# 心跳
insforge.send_heartbeat(robot_id)       # 更新 last_heartbeat
insforge.update_robot_status(robot_id, status)  # 更新状态
```

**改造要点**:
- `register()` 返回的 `robot_id`（UUID）需缓存，供后续心跳和任务拉取使用
- 心跳频率保持原逻辑不变（定期调用 `send_heartbeat()`）
- 终端 ID 映射：原 `terminal_id`（MAC 地址）→ `robots.agent_key` 字段
- 查询时通过 `agent_key` 过滤获取 `robot_id`

### 2.3 Edge Functions 业务逻辑实现

#### 2.3.1 技术模式

参考现有 Edge Functions（`notify`、`blacklist`），采用以下模式：

```typescript
// functions/robot-version/index.ts
import { createClient } from 'npm:@insforge/sdk'

Deno.serve(async (req: Request) => {
  // 1. CORS 处理
  if (req.method === 'OPTIONS') return new Response(null, { status: 204 })

  // 2. 创建 InsForge SDK client（使用请求中的 JWT）
  const authHeader = req.headers.get('Authorization') || ''
  const client = createClient({
    baseUrl: Deno.env.get('INSFORGE_BASE_URL')!,
    anonKey: Deno.env.get('ANON_KEY')!,
    edgeFunctionToken: authHeader.replace('Bearer ', ''),
  })

  // 3. 解析请求
  const { action, ...params } = await req.json()

  // 4. 业务逻辑
  switch (action) {
    case 'publish': {
      // 创建新版本记录
      const { data: version, error } = await client.database.from('workflows')
        .insert({ project_id: params.projectId, version: params.version, status: 'published' })
        .select()
        .single()
      // 通知团队成员
      await client.database.from('notification_logs')
        .insert({ type: 'in_app', recipient: params.userId, message: '新版本已发布' })
      return new Response(JSON.stringify({ success: true, data: version }), {
        headers: { 'Content-Type': 'application/json' },
      })
    }
    // ... 其他 action
  }
})
```

## 三、Fix-13: resource-service 文件管理迁移

### 3.1 前端文件上传迁移

**当前文件**: [`web-app/src/api/resource.ts`](file:///d:/develop/astron-rpa-bak/frontend/packages/web-app/src/api/resource.ts#L239-L248)

**当前调用**:
```typescript
// POST /api/resource/file/upload
export async function uploadFile(data: { file: File }, config: RequestConfig = {}) {
  const res = await http.postFormData<string>('/api/resource/file/upload', data, { timeout: 5000000, ...config })
  return res.data
}

// POST /api/resource/file/upload-video
export async function uploadVideoFile(data: { file: File }, config: RequestConfig = {}) {
  const res = await http.postFormData<string>('/api/resource/file/upload-video', data, { timeout: 5000000, ...config })
  return res.data
}
```

**迁移方案** — 使用 InsForge Storage SDK：

```typescript
// 迁移后
import { insforge } from '@rpa/shared'

export async function uploadFile(data: { file: File }, config: { bucket?: string } = {}) {
  const bucket = config.bucket || 'rpa-files'
  const file = data.file
  const path = `${Date.now()}-${file.name}`

  const { data: result, error } = await insforge.storage
    .from(bucket)
    .upload(path, file)

  if (error) throw error

  // 获取公开 URL
  const { data: { publicUrl } } = insforge.storage
    .from(bucket)
    .getPublicUrl(path)

  return publicUrl
}
```

**改造要点**:
- 在 InsForge 中创建 `rpa-files` bucket（类型 public）
- 上传后返回文件的公开 URL 而不是 fileId
- 前端需要存储文件 URL 的组件（如头像、附件）需适配新的返回格式

### 3.3 数据库 `file` 表迁移

**当前**: resource-service 依赖 MySQL 的 `rpa.file` 表，存储文件元数据（file_id, path, name, type, size, create_time, update_time, deleted, real_name 等）

**迁移方案**（两种选择）:

**方案 A：使用 InsForge Storage 元数据**（推荐）
- InsForge Storage 自带对象元数据管理
- 无需额外建表，直接通过 `storage.from().list()` 获取文件列表
- 缺点：自定义字段支持有限

**方案 B：新建 `shared_files` 表**（推荐，配合企业共享功能）

在 `001_create_rpa_tables.sql` 中追加：

```sql
-- 10. 共享文件表
CREATE TABLE IF NOT EXISTS public.shared_files (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID REFERENCES public.projects(id),
    file_name VARCHAR(255) NOT NULL,
    file_path VARCHAR(512) NOT NULL,
    file_size BIGINT DEFAULT 0,
    file_type VARCHAR(100),
    storage_bucket VARCHAR(100) DEFAULT 'rpa-shared',
    uploaded_by UUID REFERENCES auth.users(id),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    user_id UUID REFERENCES auth.users(id)
);

ALTER TABLE public.shared_files ENABLE ROW LEVEL SECURITY;

DROP POLICY IF EXISTS "Users can access their project's shared files" ON public.shared_files;
CREATE POLICY "Users can access their project's shared files" ON public.shared_files
    FOR ALL USING (auth.uid() = user_id);

CREATE INDEX IF NOT EXISTS idx_shared_files_project_id ON public.shared_files(project_id);
CREATE INDEX IF NOT EXISTS idx_shared_files_user_id ON public.shared_files(user_id);

DROP TRIGGER IF EXISTS trg_shared_files_set_user_id ON public.shared_files;
CREATE TRIGGER trg_shared_files_set_user_id
    BEFORE INSERT ON public.shared_files
    FOR EACH ROW EXECUTE FUNCTION public.set_user_id();
```

## 五、风险与注意事项

### 5.1 兼容性风险

- **Edge Functions 部署路径**: 当前 Edge Functions 通过 `Deno.serve()` 直接启动，而 MCP 的 `create-function` 工具要求 `module.exports` 格式。需确认部署方式保持一致（当前 notify 和 blacklist 是手动部署的）
- **前端返回格式**: 原 robot-service 返回 `{ code, msg, data }` 格式，迁移后需确保前端调用方兼容 InsForge SDK 的 `{ data, error }` 格式
- **文件 URL 格式**: 原 resource-service 返回 `fileId`，前端通过 `/api/resource/file/download?fileId=xxx` 下载；迁移到 InsForge Storage 后返回 `publicUrl`，前端直接使用 URL

### 5.2 权限风险

- **RLS 策略**: 确保 `shared_files` 表的 RLS 策略正确，用户只能访问自己项目下的共享文件
- **Storage bucket 权限**: `rpa-files` 设置为 public（头像、附件等），`rpa-shared` 设置为 private（共享文件需认证）

### 5.3 数据一致性

- **终端注册**: 引擎重启后应检查 `robots` 表是否已有相同 `agent_key` 的记录，避免重复注册
- **心跳超时**: 原 robot-service 有心跳超时自动离线逻辑，迁移后需在 `send_heartbeat()` 中实现类似机制，或通过 PostgreSQL 触发器实现

### 5.4 部署顺序

1. 先完成 Phase 1（引擎层，直接影响 RPA 执行）
2. 再完成 Phase 2（前端 CRUD，影响用户体验）
3. 最后 Phase 3 + Phase 4（Edge Functions，影响复杂业务逻辑）
4. 确认全部功能正常后再进入 Phase 5 清理
