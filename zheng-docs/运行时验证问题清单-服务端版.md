## 问题详情

### Issue-1: `rpa-crud:quota-check-designer` 配额检查仍返回 false

**优先级**: P0（阻塞用户创建应用）  
**状态**: ⏳ 待修复

**现象**:
- 浏览器端点击"新建RPA应用"时，前端调用 `invokeRpaFunction('rpa-crud', 'quota-check-designer', ...)`
- Edge Function 返回 `data: false`，前端弹出"已达到应用数量上限"提示
- 即使该用户一个应用都没创建也触发

**修复方案**:
- 在 `rpa-crud` Edge Function 的 `quotaCheckDesigner` 函数中直接返回 `data: true`（放行所有创建请求）
- 配额限制由前端根据 `tenantType` 控制（前端已通过 `userStore.currentTenant?.tenantType !== 'enterprise'` 判断是否需检查）
- 旧 Java 实现中 `resource_config` 表在 PostgreSQL 中不存在，无需迁移

**验证**:
- [ ] `curl` 测试返回 `{"code":200,"data":true,"message":"success"}`
- [ ] 浏览器端点击"新建RPA应用"验证正常

### Issue-12: `/api/auth/refresh` 401 (Unauthorized)

**优先级**: P2（影响会话自动续期）  
**状态**: ⏳ 待修复

**现象**:
- 浏览器控制台输出 `POST http://172.16.100.211:7130/api/auth/refresh 401 (Unauthorized)`
- 伴随 `Uncaught InsForgeError: No refresh token provided`
- 触发位置：`useUserStore.ts:31` 调用 `getCurrentUser`

**根因分析**:
- InsForge SDK 在检测到 token 过期时自动调用 `/api/auth/refresh`
- 服务端返回 401，可能是 refresh token 未存储、已过期或 InsForge Auth 配置问题

**修复方案**:
- 在 InsForge 服务端检查 Auth 配置，确认 `/api/auth/refresh` 端点正常工作
- 检查 session 中 refresh token 的存储策略
