### Issue-1: `quota-check-designer` 配额检查失败（新建RPA应用报错）

**归属**: 🖥 **服务端**  
**优先级**: P0（阻塞用户创建应用）  
**状态**: ⏳ 待修复

**现象**:
- 点击"新建RPA应用"时，前端调用 `invokeRpaFunction('rpa-crud', 'quota-check-designer', ...)` 
- Edge Function 返回 `data: false`，前端弹出"已达到应用数量上限"提示
- 即使该用户一个应用都没创建也触发

**当前实现**（`rpa-crud` Edge Function `quotaCheckDesigner` 函数）：

```javascript
// 当前实现：统计用户 robots 数量，硬编码配额 20
const { data, error, count } = await client.database.from("robots").select("*", { count: "exact" }).eq("user_id", userId);
const designerCount = count || (data ? data.length : 0);
const DEFAULT_QUOTA_LIMIT = 20;
const allowed = designerCount < DEFAULT_QUOTA_LIMIT;
```

**根因分析**:
1. 硬编码配额 20 与业务不符（专业版应为 99，企业版不限制）
2. 旧 Java 实现（`QuotaCheckServiceImpl.java`）从 `resource_config` 表读取 `designer_count` 配置，但该表在 PostgreSQL 中不存在
3. 可能的另一种原因：`count` 参数在 InsForge SDK 中可能返回的是所有记录数而非当前用户记录数，导致统计错误

**修复方案**:
- **方案A（推荐）**：简化逻辑，直接返回 `true`（放行），配额限制由前端根据 `tenantType` 控制（前端已通过 `userStore.currentTenant?.tenantType !== 'enterprise'` 判断是否需检查）
- **方案B**：在 Edge Function 中查询 `resource_config` 表获取实际配额，但需先创建该表并迁移数据
- **方案C**：修复 `count` 查询逻辑，确保只统计当前用户数据

### Issue-2: `client-sdk.js` 404 未找到

**归属**: 💻 **客户端**  
**优先级**: P1（页面加载错误）  
**状态**: ✅ 已修复（2026-07-12）

**现象**:
- 浏览器控制台报 `GET http://localhost:1420/client-sdk.js net::ERR_ABORTED 404`
- 该文件由 `electron-app` 的 SDK 构建产生，输出到 `frontend/public/client-sdk.js`

- 运行 `cd frontend/packages/electron-app && pnpm build:sdk`
- 构建成功后会生成 `frontend/public/client-sdk.js` 和 `client-sdk.js.map`

**验证**:

### Issue-4: `notify` Edge Function 调用 400 错误

**归属**: 💻 **客户端**（已修复）  
**优先级**: P2（非阻塞，仅控制台报错）  
**状态**: ✅ 已修复（2026-07-12）

**现象**:
- 浏览器控制台报 `POST http://172.16.100.211:7130/functions/notify 400 (Bad Request)`
- `taskNotify` 调用 `notify` Edge Function 时传入 `{ event: 'normal' }`，但 `notify` 函数期望 `{ type, recipient, content }`

**修复方式**:
- 将 `api-client.ts` 中 `taskNotify` 改为空函数（no-op）
- 原因：CRUD 已通过 Edge Functions 直接操作数据库，不再需要通知 trigger 服务器

**验证**:
- [ ] 重启 Vite dev server 后确认无 notify 报错

### Issue-6: Vite dev server 缓存导致 shared 包变更不生效

**归属**: 💻 **客户端**（开发习惯）  
**优先级**: P3  
**状态**: ⏳ 持续关注

**现象**:
- 修改 `shared/src/api-client.ts` 后，重建 `shared` 包（`pnpm build`），但 Vite dev server 仍提供旧版本
- 需要重启 Vite dev server 才能生效

**原因**:
- Vite 对 workspace 依赖的模块有缓存机制
- `shared` 包的 `postinstall` 钩子只在 `pnpm install` 时触发

**建议**:
- 修改 `shared` 包后，执行 `pnpm build` 然后重启 dev server
- 或使用 `pnpm dev --force` 清除 Vite 缓存
